## ✅ Auth method — DECIDED: service account + domain-wide delegation (impersonation)
Resolved 2026-06-09. The GSC UI **silently rejects service-account emails** as
property users (a SA isn't a real Google account), so direct-add is impossible.
Instead the SA uses **domain-wide delegation** to **impersonate** a Workspace user
who already owns the properties.

- Service account: `search-console-agent@golfvilla-search-console.iam.gserviceaccount.com`
  (GCP project `golfvilla-search-console`, client_id 115283306924568576295).
- **Domain-wide delegation** authorized in Google Workspace Admin → Security → API
  Controls → Domain-wide Delegation: client ID `115283306924568576295`, scope
  `https://www.googleapis.com/auth/webmasters.readonly`.
- **Impersonation subject:** `rbender@fortisnetlease.com` — the Workspace user with
  Owner access to BOTH `golfvilla.com` and `espadavilla.com`. The SA acts AS this
  user. (The SA email itself is NOT added to the properties.)
- Code builds a JWT from the decoded key with `scopes=[webmasters.readonly]` and
  `subject=GSC_IMPERSONATED_SUBJECT`. Without `subject`, the SA has no GSC access.
- Key delivery: the full key JSON is **base64-encoded** into `GSC_SERVICE_ACCOUNT_JSON`
  (gitignored `.env.local` + Vercel Sensitive). The raw .json is never committed
  (`.gitignore` blocks `*-search-console-*.json` etc.).

### Env vars (service-account + impersonation model)
- `GSC_SA_EMAIL` — service-account email (non-secret; reference).
- `GSC_IMPERSONATED_SUBJECT` — Workspace user to impersonate (e.g. `rbender@fortisnetlease.com`).
- `GSC_SERVICE_ACCOUNT_JSON` — **secret**; base64 of the key JSON. Rob pastes it.
- `GSC_SITE_URL` — e.g. `sc-domain:golfvilla.com`.
